ARCZT Vault — Features

Zero-knowledge credential management. Your safe never leaves your device.

🔐 Zero-knowledge
Your safe is encrypted with keys derived from your master password. ARCZT cannot decrypt your data — we don't have your key and we don't want it.

Feature Overview

Encrypted Safe

All entries in your vault — passwords, API keys, secure notes — are encrypted at rest using AES-256-GCM (authenticated encryption). The encryption key is derived from your master password using Argon2id, a memory-hard key derivation function that resists GPU and ASIC brute-force attacks.

Each entry is encrypted individually, not just the safe as a whole. This means that even if an attacker obtained partial access to the safe file, they would still need to decrypt each entry independently. The safe file includes a per-entry nonce and authentication tag, ensuring that any tampering is detected on read.

Parameters: Argon2id with 64 MB memory cost, 3 iterations, 4 parallel lanes, 16-byte random salt. AES-256-GCM with 96-bit nonces. These parameters exceed current OWASP recommendations for password hashing.

Password Generator

ARCZT includes a cryptographically secure password generator. It uses the platform's CSPRNG (Cryptographically Secure Pseudo-Random Number Generator) — SecRandomCopyBytes on Apple platforms, java.security.SecureRandom on Android — not Math.random() or any predictable source.

You can configure:

  • Length — 8 to 128 characters (default: 24)
  • Character sets — uppercase, lowercase, digits, symbols (each toggleable)
  • Strength meter — Real-time entropy estimation in bits, with color-coded feedback (weak, fair, strong, very strong)
  • Excluded characters — Optionally exclude ambiguous characters (l, 1, O, 0, I) to reduce transcription errors

Generated passwords never touch disk. They exist only in memory until you copy them to the clipboard (which is cleared after 30 seconds) or save them to an entry.

MFA & SSO

ARCZT Vault supports multi-factor authentication for vault access. In addition to your master password, you can enable a second factor:

  • TOTP — Time-based One-Time Passwords (RFC 6238). Scan a QR code to enroll, then enter the 6-digit code from any authenticator app (Authy, Google Authenticator, 1Password, etc.) at unlock.
  • Biometric — Face ID, Touch ID, or fingerprint as a second factor (in addition to or instead of TOTP for re-unlock after auto-lock).
  • Hardware key — FIDO2/WebAuthn security keys (YubiKey, Titan) for hardware-backed second factor.

SSO integration allows organizations to use single sign-on providers (OIDC, SAML) for initial authentication, after which the master password derives the local encryption key. SSO does not replace the master password — it adds an identity layer on top of the zero-knowledge encryption.

Credential Sharing

ARCZT allows you to share safe entries with other users using end-to-end encryption. When you share an entry, ARCZT:

  • Generates a one-time symmetric key (AES-256)
  • Encrypts the entry with that key
  • Encrypts the key with the recipient's public key (RSA or ECDH key exchange)
  • Produces a shareable link containing the encrypted payload

The recipient decrypts the key with their private key, then decrypts the entry. The sharing link can be time-limited (expiry from 1 hour to 7 days) and can be revoked at any time by the sender. After expiry or revocation, the link is invalid and the entry cannot be decrypted.

ARCZT's servers (if using hosted mode) only relay the encrypted payload — they cannot decrypt the entry. In self-hosted mode, the relay server runs on your own infrastructure.

Audit Log

ARCZT maintains a tamper-evident audit log of all activity within the vault. Each log entry is chained to the previous one using a cryptographic hash, so any modification or deletion is detectable.

The audit log tracks:

  • Entry creation — When an entry is added, with timestamp and entry metadata (title, domain — not the credential itself)
  • Access events — When an entry is viewed, copied, or autofilled, with timestamp
  • Modifications — When an entry is edited, with before/after field hashes (not the actual values)
  • Deletions — When an entry is removed, with timestamp and a hash of the deleted entry
  • Sharing events — When an entry is shared or a share is revoked

The audit log itself is encrypted at rest and can only be read with the master password. Export to CSV or JSON is supported for compliance and forensic review.

Breach-Aware

ARCZT links safe entries to known data breaches through the Vigilar bridge. When Vigilar detects a breach involving a domain that matches an entry in your safe, it sends breach metadata (domain, breach name, date, severity) to ARCZT via the deep-link bridge.

ARCZT then highlights the affected entries in your vault with a warning indicator. You can see which credentials are exposed, when the breach occurred, and take action — typically generating a new password for the affected service and updating the entry.

No credentials flow between Vigilar and ARCZT. The bridge is metadata-only. Vigilar does not read your vault, and ARCZT does not send any entry data back to Vigilar.

Browser Extension

The ARCZT browser extension provides autofill from your vault without exposing the safe. The extension communicates with the ARCZT desktop app over a local IPC channel (not over the network). When you navigate to a login page, the extension requests credentials for that specific domain.

Per-domain isolation ensures that each website can only access credentials intended for its domain. A credential for example.com cannot be read by attacker.com. The extension verifies the current URL against the credential's associated domain before offering autofill, and the user must confirm each autofill action.

The extension never receives the decrypted safe — it receives only the specific entry needed for the current page, and only after user confirmation. No credentials are cached in browser storage. The extension supports Chromium-based browsers (Chrome, Edge, Brave) and Firefox.

Mobile Apps

ARCZT Vault has native mobile apps for Android and iOS:

  • Android — Written in Kotlin, using Jetpack Compose for UI and the platform Keystore for hardware-backed key storage. Supports biometric unlock via fingerprint and face unlock.
  • iOS — Written in Swift, using SwiftUI for UI and the Secure Enclave for hardware-backed key storage. Supports Face ID and Touch ID for biometric unlock.

Both apps use the same encryption parameters (AES-256-GCM, Argon2id) as the desktop app. The safe file format is cross-platform — you can move a safe between desktop and mobile by importing the encrypted backup file. Each platform derives the key locally using its own native crypto APIs, but the resulting encryption is identical.

The mobile apps do not sync to any cloud. They are fully offline, just like the desktop app. Transferring a safe between devices is done by exporting an encrypted backup and importing it on the target device.
