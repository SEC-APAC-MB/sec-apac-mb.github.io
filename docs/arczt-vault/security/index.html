ARCZT Vault — Security Model

Zero-knowledge encryption. Your master password never leaves your device. We can't decrypt your data — we don't have your key.

🔐 Zero-knowledge by design
ARCZT Vault is built so that your encrypted safe can only be opened with your master password. The master password is never stored, never transmitted, and never processed on any server.

Zero-Knowledge Architecture

ARCZT Vault uses a zero-knowledge architecture. Your master password is used solely to derive the encryption key that unlocks your safe. It is never stored on disk, never transmitted over any network, and never sent to ARCZT's servers (or anyone else's). The key derivation and encryption/decryption all happen locally on your device.

This means that even if ARCZT's servers were compromised, an attacker would find only encrypted blobs that cannot be decrypted without your master password. We don't have your key, and we don't want it.

Encryption

All safe entries are encrypted with AES-256-GCM (Galois/Counter Mode), which provides both confidentiality and integrity (authenticated encryption). Any tampering with ciphertext is detected on decryption.

The encryption key is derived from your master password using Argon2id, a memory-hard key derivation function designed to resist GPU and ASIC brute-force attacks. The parameters are:

  • Memory: 64 MB per derivation — forces attackers to use significant RAM per guess
  • Iterations: 3 — number of passes over the memory buffer
  • Parallelism: 4 — number of parallel lanes for computation
  • Salt: 16 bytes, cryptographically random, unique per safe — prevents rainbow table attacks

Argon2id was selected over PBKDF2 and bcrypt because it is resistant to both time-memory trade-offs and side-channel attacks, and is the recommended password hashing algorithm by the Password Hashing Competition (PHC).

Quantum Resistance

ARCZT Vault's symmetric encryption (AES-256) is considered quantum-resistant. Grover's algorithm, the most relevant quantum attack against symmetric ciphers, provides a quadratic speedup — effectively reducing AES-256's security to 128-bit against a quantum adversary.

128-bit security is still considered infeasible to break with current and near-future quantum computers. For context, brute-forcing 128 bits would require approximately 2^128 operations, which is beyond the capability of any projected quantum computer.

The Argon2id key derivation is also quantum-resistant: memory-hard functions are not meaningfully accelerated by quantum computation. The key derivation cost is dominated by memory access patterns, not raw computation, which quantum computers do not speed up.

Auto-Lock & Biometrics

ARCZT Vault auto-locks after a configurable idle timeout (default: 5 minutes). When locked, the decrypted key is purged from memory and the safe becomes inaccessible without re-entering the master password.

For convenience, ARCZT supports biometric re-unlock (Face ID on macOS/iOS, fingerprint sensor on Android). When biometric unlock is enabled, the master password is used to derive a key that is stored in the platform's secure enclave (Secure Enclave on Apple Silicon, StrongBox on Android). Biometric authentication unlocks this enclave-stored key — your master password itself is never stored, only the derived key, and only in hardware-backed secure storage.

You can configure the idle timeout from 1 minute to 30 minutes, or set it to "never" (not recommended). Biometric unlock can be toggled independently of the auto-lock timer.

Network Isolation

ARCZT Vault operates entirely offline. There is no telemetry, no sync, no cloud backup, and no account requirement. The safe is local only — it lives in an encrypted file on your device's storage.

This means there is nothing to intercept. No credentials are transmitted over any network. No metadata about your safe entries (domains, usernames, timestamps) is ever sent anywhere. The safe cannot be accessed remotely because it has no network presence.

Backups are your responsibility — and by design. You can export an encrypted backup of your safe at any time. The backup file is encrypted with the same AES-256-GCM + Argon2id parameters as the live safe, so it can only be restored with your master password.

Browser Extension Security

The ARCZT browser extension can autofill credentials from your vault without exposing the full safe. The extension communicates with the ARCZT desktop app over a local IPC channel (not over the network). When you navigate to a login page, the extension requests credentials for that specific domain.

Per-domain isolation ensures that each website can only access credentials intended for its domain. A credential for example.com cannot be read by attacker.com. The extension verifies the current URL against the credential's associated domain before offering autofill.

The extension never receives the decrypted safe — it receives only the specific entry needed for the current page, and only after user confirmation. The extension does not cache credentials in browser storage.

Vigilar Bridge Security

ARCZT connects to Vigilar through a one-directional deep-link bridge. When Vigilar detects a breach, it sends a deep-link notification to ARCZT containing only breach metadata: the affected domain, breach name, breach date, and severity level.

No credentials flow between the two apps. Vigilar does not read your vault contents. ARCZT does not send any safe entries to Vigilar. The connection is strictly one-way: Vigilar pushes breach metadata, ARCZT receives it and matches it against local entries.

The deep-link bridge uses the platform's native URL scheme handling (arczt:// on desktop, intent URIs on Android). No shared database, no shared filesystem, no shared process. The two apps are fully isolated except for this narrow metadata-only channel.
