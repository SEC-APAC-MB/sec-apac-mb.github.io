Vigilar — Security Model

No cloud. No telemetry. No background services. Your data never leaves your device.

🔒 On-device architecture
Vigilar is designed around a simple principle: your security data should never leave your device. Every feature — AI chat, threat detection, breach checks — runs locally or uses privacy-preserving protocols.

On-Device Architecture

Vigilar runs entirely on your device. The AI companion model (Ember Prime v8c, 12B parameters, Q4_K_M quantization) loads into your device's RAM at startup and performs all inference locally via llama.cpp on desktop and MLC on mobile. No network connection is required after installation — the model, threat signatures, and breach-check logic are all bundled.

The only network call Vigilar makes is the breach check, which contacts the Have I Been Pwned API using a k-anonymity protocol (see below). Even this call does not transmit your email address in a form that could identify you.

Data Persistence

Vigilar stores only a license heartbeat — a small validation token used to confirm your license is active. This token is stored securely using SecureStore on Android and Keychain on macOS. No chat conversations are persisted to disk. No scan results are stored in localStorage or any cloud database. There is no cloud sync of any kind.

When you close Vigilar, your chat history is gone. When you reopen it, the model reloads fresh. This is intentional — there is nothing to steal, because there is nothing stored.

Breach Check Privacy

Breach checks use the Have I Been Pwned (HIBP) API with k-anonymity hashing. Here's how it works:

  • Vigilar computes the SHA-1 hash of your email address locally on your device.
  • Only the first 5 characters of that hash are sent to HIBP — not your email, not the full hash.
  • HIBP returns all breach entries that match that 5-character prefix (approximately 200 hashes).
  • Vigilar filters the returned list locally to find matches for your full hash.
  • Your full email address and full hash never leave your device.

HIBP never knows which specific email you're checking. They only see that someone with a hash starting with those 5 characters made a request. This is the same k-anonymity model used by major password managers and is considered the gold standard for breach-check privacy.

AI Model Security

The AI companion is powered by Ember Prime v8c, a 12B parameter model custom-trained by SEC-APAC-MB. The model is quantized to Q4_K_M format (approximately 7.4 GB) and runs in Vigilar's process memory. It does not make any external API calls.

The model has no network access. It cannot phone home, exfiltrate data, or communicate with any server. All prompts and responses exist only in your device's RAM during the active session and are discarded when the app closes. The model binary is included in the app bundle and loaded locally at startup.

Ember Prime v8c was trained with governance and safety constraints. It includes a fingerprint layer that identifies it as an SEC-APAC-MB model and a governance LoRA that biases responses toward security best practices. The model will not generate instructions for exploiting vulnerabilities or crafting malware.

License Verification

Vigilar uses a lightweight license verification system. On launch, it sends a heartbeat to Stripe's license validation endpoint containing only your license key. Stripe confirms whether the key is valid and active. The result is cached locally for a configurable period (default: 7 days), so Vigilar can operate offline between checks.

There is no always-online DRM. If the heartbeat fails (e.g., you're offline), Vigilar continues to operate using the cached validation result. Only the license key is transmitted — no usage data, no chat content, no scan results.

Attack Surface

Vigilar is designed to minimize its own attack surface:

  • No listening ports — Vigilar does not open any network ports. It does not run a web server, API server, or any service that accepts inbound connections.
  • No background services — Vigilar does not run as a daemon or background service. It is active only when the app is open. When closed, it consumes no CPU and has no network presence.
  • No telemetry — Vigilar does not collect, aggregate, or transmit usage analytics. There are no tracking pixels, no crash reporters that send data without consent, and no usage counters.
  • No third-party SDKs — Vigilar does not include third-party analytics or advertising SDKs. The only external dependency is the HIBP API for breach checks.

Threat Model

What Vigilar protects against:

  • Misconfigured macOS security controls (disabled Gatekeeper, SIP, FileVault, Firewall)
  • Known data breaches affecting your email address
  • Suspicious processes and persistence mechanisms (launch agents, daemons)
  • Unauthorized open ports and network services
  • Outdated or vulnerable installed applications
  • Weak identity theft posture (reused passwords, plaintext credentials)

What Vigilar does not protect against:

  • Zero-day exploits with no known signatures or behavioral patterns
  • Kernel-level rootkits that operate below the process monitoring layer
  • Physical device compromise (evil maid attacks) — use FileVault for disk encryption
  • Network-level attacks — use a firewall and VPN for network protection
  • Social engineering — the AI companion can advise, but it cannot prevent you from voluntarily giving away credentials

Vigilar is a detection and advisory tool. It is not a replacement for EDR (Endpoint Detection and Response) solutions in enterprise environments. For personal devices and small teams, Vigilar provides strong baseline protection and security awareness.
